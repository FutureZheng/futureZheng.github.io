零拷贝技术

2021年6月3日
| 操作系统 消息队列

一。 一次简单的文件读写操作。

今天来简单聊一聊文件的读取操作。

也许你在日常开发当中，难免需要涉及到文件的操作，比如读取一个文件的内容，然后应用程序再对内容做相关的处理，然后再发送给用户。 这么一个简单的操作，需要涉及到系统的调用，比如

read(fd, buf)
write(fd, buf)

下面展示了传统io的操作。

从上面可以看到，当用户进程需要读取系统内的文件内容，需要用户进程发起系统调用比如read函数，读取磁盘的数据内容，这个时候，操作系统控制cpu,将当前进程的运行模式，从用户空间切换到 内核空间，并且发送一条指令到DMA上，然后操作系统会控制cpu，挂起当前用户进程，然后从就绪队列中获取下一个执行的用户进程来运行。再cpu运行其他进程的情况下，DMA一直持续工作着，将 磁盘的数据copy到内核的pagechache中，待copy完毕，发出一个中断信号，中断处理函数告知cpu处理之前被挂起的进程，cpu从进程控制块（PCB）中加载之前挂起程序的寄存器和栈信息，并继续 执行程序，cpu控制将pagecache的数据copy到用户空间上，read函数返回。

可以看出来，一次系统调用（比如read），涉及到了2次的数据copy和2次的上下文切换。

数据copy

1. DMA拷贝，从磁盘缓冲区数据cop到内核缓冲区pageCache中。
2. cpu拷贝，从内核缓冲区pagecache的数据到用户进程缓冲区中。

上下文的切换：

发起一次系统调用，需要将进程从用户空间切换到内核空间。
函数返回，又要将进程从内核空间切换到用户空间上。

用户进程调用系统函数write,将数据写入到网卡设备上的操作跟read是反过来的。

综上所述，一次完整的io操作，需要涉及到4次的数据copy,4次的上下文切换。

二。 零copy技术

  1. 什么场景下能使用零copy技术？

观察上面的数据读取和发送过程，可以发现几个问题点。

  • 本质上是用户进程需要对copy过来的数据，进行二次加工，才需要将数据copy到用户进程里面。
  • 如果存在场景，不需要用户进程对copy的数据进行二次加工，那么本质上就没必要把数据copy到用户进程上。

所以可以看出。零copy的技术，适用于不需要对数据做二次加工的场景下，比如作为静态资源服务器等

  1. 什么是零copy技术？

零copy技术，可以指真正没有进行数据copy,但是我认为零copy技术，指的是减少数据copy次数。 要想减少数据copy的次数，本质上就是减少系统函数的调用。一次系统函数的调用，就涉及到2次上下文切换和两次的数据copy.

常见的零copy技术

  1. mmap + write
  2. sendfile

1） mmap + write的方式

当用户进程调用系统函数mmap后，会在内核缓冲区和用户进程缓冲区建立一个共享区域，这样子可以减少了一次数据拷贝的次数来提高数据传输的效率。

从上面可以看出来

  1. 用户进程第一次调用mmap的时候，会在内核空间和用户进程空间，共享一块存储区域，所以调用mmap需要2次上下文的切换和1次数据copy(从磁盘copy到pagecache)
  2. 当系统调用write的时候，cpu会将pagechche的数据copy到socket缓冲区上面，然后再通过DMA将数据copy到网卡缓冲区上。这就涉及到一次cpu拷贝，一次DMA数据copy.

综上所述，使用mmap + write的方式，涉及到3次的数据copy,因为需要进行两次系统函数的调用，所以，还是需要4次上下文的切换。

2） sendfile的方式

通过上面mmap + write方式的分析，可以看出来，虽然减少了一次数据copy的方式，但是还是有多次的上下文切换的开销，因为每次调用系统函数的时候，都需要进行用户空间和内核空间的切换 那么有没有办法减少上下文切换的次数呢？（本质上就是减少系统函数调用的次数）

这个时候就需要使用sendfile了，来看一下sendfile的整个过程。

先看下函数sendfile的实现

NAME
        // 再两个文件描述符之间传输数据
       sendfile - transfer data between file descriptors
SYNOPSIS
       #include <sys/sendfile.h>
       ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);

从上面可以看出来

  1. 用户简称调用sendfile函数，可以让cpu将pagechche的数据直接copy到socket缓冲区上面，这样子可以直接再内核中完成数据copy,而无需再由用户进程发起函数调用。

综上所述，数据copy的次数同mmap + write的方式，上下文的切换，由于减少了系统函数的调用，所以上下文切换变成2次，数据copy的次数还是3次。

3）你以为这样子就完了？？？

当然不是，如果你的网卡支持（The Scatter-Gather Direct Memory Access）即 SG-DMA的话，那么再数据参数上面，可以不需要经过socket的缓冲区，即不需要cpu进行一次数据copy.

整个过程，仅涉及到一次sendfile函数的调用，以及DMA的数据copy,无需cpu参与进行数据copy，所以上下文切换为2次，数据copy次数为2次。

三。零copy技术使用的注意项

上面使用零copy的方案上，不管是哪一种方式，其实都是需要使用到pagechae（磁盘高速缓存），如果使用大文件的传输使用零拷贝技术的话，那pagecache很快就会被文件的数据占满，导致小数据的读取没办法得到速度上的提升， 所以大文件的传输，需要尽量避开pagecache的方式。

  • 针对小文件，可以采用零copy的技术加快文件的传输
  • 针对大文件，可以采用异步IO + 直接IO
