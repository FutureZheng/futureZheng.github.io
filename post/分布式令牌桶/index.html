分布式限流-分布式令牌桶

2021年7月26日
| 分布式系统

一。单体应用带来的问题？

之前说到了单机令牌桶的使用，实现上面来说相对简单，但是实际项目开发的时候，考虑项目扩展，都会采用部署多个应用，比如部署多个web项目，或者将应拆分为多个微服务的方式进行部署。

这就要求了，如何确保多个应用请求总和加起来不超过限制。

由于多个应用可能部署在不同的进程上面，或者机器上面，所以没办法跟传统的单机应用一样，使用基于进程的rate,因为数据不共享。

所以要求，需要有一个统一的应用，来统一管理所有web应用的请求记录信息等。

二。分布式令牌桶

类似于实现下面的这种方式，所有的请求，都需要经过统一应用来确认是否存在令牌信息。

思考一下，这种方式是不是跟api gateway的方式是一致的呢？

没错，可以在api gateway上面进行流控。

1.基于redis实现的分布式令牌桶

由于redis在处理命令的时候，是单进程的方式，所以天生能保证原子性操作。

并且操作redis实现的令牌桶，涉及到多个命令，也需要保持它的原子性，所以这边普遍的做法是redis + lua的方式。

这边我给出了我简单实现的代码

-- 获取 key 和 令牌桶的大小
local key = KEYS[1]
local limit = tonumber(ARGV[1]) or 0

-- 获取redis中存储的limit,如果没有，返回0
local currentLimit = redis.call("get", key) or 0

-- 已经超过令牌桶的限制
if currentLimit + 1 > limit then
    return 1
end

-- 重新设置令牌桶信息
redis.call("incr", key)
redis.call("expire", key, 1)

return 0

这边简单的硬编码了1s的过期时间，更好的方式是通过外部传入进来，业务按照各自的需要来设置。

依旧采用jemeter来验证下。

result, err := redis.Int(redis.NewScript(1, LuaScript).Do(conn, "rate-key", "3"))

这边设置了令牌桶的大小为 3, 增长的步长就是lua脚本,步长为1

1. 单个进程验证

使用jemeter用同样的规则验证下

看对应的效果

总共请求10次，设置了令牌桶的大小为3，1s的过期时长，所以，展现出来的就是每一秒钟可以处理3个请求，符合预期。

2.多个进程的情况

这边我本地安装了kubernetes，单个节点，我这边用相同的服务，启多个进程，来验证下。（这边我提前安装了一个redis的服务）

符合设定的了，令牌桶的大小为 3, 增长步长为1的情况，并且多个进程也不会出现问题的。

3.这样子就完了吗？

我后续考虑了一些情况，比如说，

  1. 为了更好的适应每个接口针对令牌桶时间和总数的配额，还需要将增长步长和key的过期时间暴露出来给业务方。
  2. 目前我这边接口的限流是放在业务代码上面，是否可以提供一个通用的业务网关来处理会比较合适呢？
4. 备注信息

这边再使用 redis-cli 执行脚本的时候，再传递KEYS，ARGV两个都是可变参数的时候，redis-cli 命令，使用逗号隔开,并且前后额外留一个空格。

redis-cli --eval rateLimiter.lua rate-key , sadas
