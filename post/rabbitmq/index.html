消息中间件-rabbitMq

2020年8月10日

为什么使用队列，也是老生常谈的问题了，异步，削峰，解耦。

由于最近一直在看rabbitMq相关的内容，加上工作中，也是采用的rabbitMq作为消息中间件的，在闲暇之余写下了这篇文章，也是对最近学习的一次总结。

谈及rabbitMq，那就要先说到AMQP协议了，简单的说来，AMQP就是一套消息通讯协议的标准。比如早期的ActiveMq,和现在rabbitMq。

先说下为什么选型的时候用的rabbitMq?

当然，这个基于下面三点考虑，

第一：公司产品本身的并发量不够，采用rabbitMq基本满足日常的使用。

第二：对比其他的mq产品，比如activeMq市场和活跃度不够，rocketMq是采用java开发的，公司大部分是golang开发的，kafka相对来说运维和学习成本比较大。

第三：学习成本低，开发的水平不一致。

RabbitMq的架构图

AMQP一些相关概念

producer: 生产者，就是生产消息的。

channel: 信道。每一个线程一个信道，多个信道可以复用tcp连接。因为操作系统上，tcp的连接是有限的65535个，并且创建和销毁tcp的开销大。

broker: 代理服务。

vhost: 虚拟主机。rabbitMq可以创建不同的虚拟主机，来隔离broker服务.

exchange: 交换机。rabbitMq的有四种交换机，direct,fanout,topic和headers

  1. direct：直连交换机。完全匹配路由键(routingKey)到绑定的队列上面。
  2. fanout：群发交换机。广播的方式发送的绑定的队列。
  3. topic： 主题交换机。通过配合通配符(*和#)，来发送消息到指定(routingKey)的交换机上面，

bindingKey: 绑定键。声明交换机和队列的绑定规则。

routingKey: 路由键。声明消息经过交换机到队列的路由规则。

queue: 队列。用于实际存储消息的位置。队列有其默认的大小，可以通过客户端的参数进行设置(x-max-length),当队列达到对应的最大值限制以后，队列会会拒绝接受消息。

consumer: 消费者。用于消费队列中的消息。

golang 连接rabbitMq服务

生产者代码：

// 建立连接
conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
if err != nil {
    log.Fatalf("amqp dial err:%s", err.Error())
}
defer func() {if err := conn.Close(); err != nil{log.Fatal(err)}}()

// 创建一个channel
ch, err := conn.Channel()
if err != nil {
    log.Fatalf("conn channel err:%s", err.Error())
}
defer func() {if err := ch.Close(); err != nil{log.Fatal(err)}}()

// 手动生命一个交换机
exchangeName := "HELLO.WORLD"
err = ch.ExchangeDeclare(exchangeName, amqp.ExchangeTopic, true, false, false,false,nil)
if err != nil{
    panic(err)
}

// 声明需要发送消息的队列
queueName := "hello"
q, err := ch.QueueDeclare(queueName, true, false, false, false, nil)
if err != nil {
    log.Fatalf("ch queuedeclare err:%s", err.Error())
}

// 声明一个绑定键,并且绑定交换机和队列
routingKey := "hello.#"
err = ch.QueueBind(queueName,routingKey,exchangeName, false, nil)
if err != nil{
    panic(err)
}

body := "hello,world"
// 发送消息到队列里面
err = ch.Publish(
    exchangeName,     // exchange
    q.Name, // routing key
    true,  // mandatory
    false,  // immediate
    amqp.Publishing{
        ContentType: "text/plain",
        Body:        []byte(body),
        DeliveryMode: amqp.Persistent,			// 持久化消息
    })
fmt.Println(err)

相关参数的说明:

声明交换机/队列：

durable: 用来确认当前的交换机，队列是否需要持久化。durable = true 表示进行持久化操作。

声明消息：

mandatory：当消息经过交换机，根据绑定规则以后，找不到指定的队列后，如何处理消息，一种是直接丢弃消息(mandatory = false)。另外一种是将消息的信息打回去给生产者(mandatory = true)。
deliveryMode: 如果 deliveryMode = 1，表示消息不进行持久化，deliveryMode = 2 表示消息持久化。

说明：正常我会将交换机和队列的创建，绑定等操作放在rabbitMq提供的管理界面上。程序上面，只进行消息发放的简单操作。

如何保证消息的可靠性传输

先看下，一条消息的所有生命周期

  1. 生产者：由生产者生成消息，经过信道到达指定虚拟主机下面的交换机，交换机通过路由键，查询绑定规则，分发到指定的队列上面。
  2. 消费者：消费者通过信道建立连接，声明要监听的队列名称。当生产者生产了消息以后，消费者就会收到消息。

其中经过4个路径，如何确保消息不丢失？

  1. 生产者到broker:
    1. 事务的方式，同步的方式，会影响rabbitMq的性能。
    2. confirm方式：异步的方式，通过生产者确认rabbitMq发送过来的ack或者nack消息，来处理消息是否重发操作。
  2. 经过exchange： 持久化交换机，从内存持久化到硬盘上。
  3. exchange到达queue: 持久化队列和消息。
  4. queue到消费者: 手动ack方式。消费者关闭自动ack的方式，改用手动ack,当消费者正常发送消费完毕以后，需要发送ack确认告诉消息队列，消息队列收到ack确认会自动删除消息和持久化消息。

这边引申出来几个问题？

  1. 如果消费者发送的ack确认到rabbitMq中，确认消息丢失了，rabbitMq是怎么处理的？
  2. 当已经消费过的消息，再次发送到消费者的时候，消费者要怎么处理呢？
  3. 如果消费者一直消费不了消息，发给rabbitMq为nack,那rabbitMq会一直重新发送消息，变成死循环，怎么处理？
  4. 多个消费者监听同一个队列，rabbitMq是怎么处理消息的呢？
